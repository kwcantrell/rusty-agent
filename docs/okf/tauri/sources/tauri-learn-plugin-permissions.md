# Summary

## Overview

This guide teaches developers how to enable and disable plugin permissions in Tauri applications, understand default permission sets, and customize permissions for specific use cases.

## Core Concepts

**Default Permissions**: Each plugin includes a `default` permission set that provides reasonable baseline access. For official plugins, these are documented in the Tauri docs. Community plugins define defaults in `your-plugin/permissions/default.toml`.

**Permission Structure**: The filesystem plugin example shows that default permissions can include:
- Command enablement (e.g., `"read-all"`)
- Scope definitions (e.g., `"scope-app-recursive"`)
- Denials (e.g., `"deny-default"`)

The default fs permissions grant all read operations and access to the `$APP` folder while denying access to sensitive webview data on Windows.

## Security Model

Permissions are configured through **capabilities** files in `src-tauri/capabilities/` or directly in `tauri.conf.json`. A capability specifies which windows can access which permissions.

Custom scopes restrict commands to specific paths. Rather than granting blanket `allow-write-text-file` access, developers should scope it precisely:

```json
{
  "identifier": "fs:allow-write-text-file",
  "allow": [{ "path": "$HOME/test.txt" }]
}
```

This follows the principle of least privilege—only exposing necessary filesystem access.

## Workflow

1. Add plugins via `tauri add <plugin-name>` or manually through Cargo
2. Review default permissions in documentation or source code
3. Identify required commands from autogenerated permission tables
4. Define custom scopes for file/directory access
5. Add permissions to the capability configuration
6. Test functionality and verify no unauthorized access errors occur

Attempting operations without proper permissions triggers errors like: `"fs.write_text_file not allowed"`
