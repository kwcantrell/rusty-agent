# Release pipeline and auto-updates

Distribution has two halves that reinforce each other: a CI pipeline that builds
and releases signed artifacts across platforms, and an updater that delivers new
versions to installed apps. Tauri's tooling covers both — `tauri-action` for the
pipeline and the updater plugin for in-place updates — and the updater's signing
requirement ties directly back into how you run the pipeline.

## Multi-platform builds with tauri-action

The primary CI tool is `tauri-action`, which automates building and releasing
across platforms; it can even auto-initialize Tauri config on a repo that lacks it
[1]. The build itself is `tauri build`, which compiles and bundles; for
customization you can split the steps (`--no-bundle`, then `tauri bundle --bundles
app,dmg`) [2]. Version comes from `tauri.conf.json > version`, falling back to the
`Cargo.toml` package version if unset [2].

A typical workflow checks out the repo, installs Linux build dependencies
(`libwebkit2gtk-4.1-dev`, `libappindicator3-dev`, `librsvg2-dev`, `patchelf`,
`xdg-utils`), installs Node LTS and a Rust toolchain (both with caching), builds
the frontend, and runs `tauri-action` [1]. Trigger on a push to a `release` branch
or on git tags like `app-v*` [1]. Build a platform matrix — macOS ARM and x64,
Ubuntu x64 and ARM64, Windows x64 — with per-platform build args [1]. Two
practical snags: the workflow needs *write* permissions on the GitHub token to
create releases (configure under Actions > Workflow permissions) [1]; and for
Apple Silicon builds downloaded from GitHub releases, use ad-hoc signing to avoid
macOS treating them as "damaged" [1]. Native ARM64 uses `ubuntu-22.04-arm` runners
on public repos; private repos fall back to slower emulation via
`pguyot/arm-runner-action` [1]. Set up real signing per the platform guides
([/practices/code-signing.md](/practices/code-signing.md)) so releases ship signed
[1].

## The updater plugin and its mandatory signatures

The updater plugin updates an app from an update server or a static JSON [3]. Full
support covers Windows, Linux, and macOS; Android and iOS have lower support;
minimum Rust is 1.77.2 [3]. The non-negotiable design point is that **signature
validation is mandatory and cannot be disabled** [3]. It uses a public/private key
pair: the public key (generated by the Tauri CLI, embedded in the binary at compile
time) validates artifacts before install; the private key signs installers and must
stay confidential [3]. Losing the private key means you can never publish another
update to existing installs — so guard it like any release-signing key, and supply
it to builds via an environment variable, not a `.env` file [3].

Configuration needs three things: the public key (the key string itself — "It
**cannot** be a file path!"), the update endpoints (URLs supporting the
`{{current_version}}`, `{{target}}`, `{{arch}}` variables), and
`createUpdaterArtifacts: true` to emit update bundles during builds [3]. The server
side comes in two shapes: a **static JSON** listing version (SemVer), a platforms
object keyed by `OS-ARCH`, the signature, and an update URL; or a **dynamic
server** that returns HTTP 204 when no update is available and 200 with the update
JSON otherwise [3]. On Windows, choose an install mode — `passive` (default, with a
progress bar), `basicUi` (requires interaction), or `quiet` (no feedback, limited
privilege escalation) — and note that Windows automatically exits the app before
installing [3].

Because the updater's signing key is separate from OS code-signing certificates,
plan for *two* secrets in the pipeline: the platform signing certificate
([/practices/code-signing.md](/practices/code-signing.md)) and the updater private
key — both injected as CI environment secrets so releases are both OS-signed and
updater-signed [1][3].
